# Documentation Review: `src/ham/bio/data.py`

**Reviewer:** Doc Reviewer Agent  
**Date:** 2026-05-15  
**Spec Version:** 1.1.0 (Berwald Revision)  
**Source:** [src/ham/bio/data.py](src/ham/bio/data.py)

## Summary

Overall documentation quality: **needs work**.

The file provides two public symbols — `BioDataset` (a `NamedTuple`) and `DataLoader` — forming the single-cell biology data ingestion layer of HAMTools. `BioDataset` is re-exported from [src/ham/bio/\_\_init\_\_.py](src/ham/bio/__init__.py), while `DataLoader` is not (but is consumed directly in several example scripts). Documentation is sparse: there is **no module docstring**, **no class docstring on either public symbol**, and only one of five methods has any docstring at all (`extract_lineage_pairs`, with a brief priority summary but no `Args`/`Returns`/`Raises`). The only private method (`_extract_pseudotime_pairs`) is entirely undocumented.

Key concerns:

1. **Terminology mismatch with spec:** `spec/ARCH_SPEC.md` § 5 describes this file as providing *"lineage triples"* and `spec/ARCH_SPEC.md` § 6.4 references *"lineage-triple batching"*. The implementation uses *"lineage pairs"* (two-element `[parent, child]` arrays). The docstring and field name `lineage_pairs` do not acknowledge this discrepancy.
2. **`BioDataset` fields are undocumented.** ML engineers have no guidance on shapes, dtypes, or semantics of `X`, `V`, `labels`, `lineage_pairs`.
3. **`DataLoader` documents no method signatures.** AnnData preprocessing, clone extraction, and pseudotime heuristics are complex pipelines that are invisible to users who only read docstrings.
4. **No examples or cross-references** in any docstring.

## Spec Alignment Notes

1. **"lineage triples" vs "lineage pairs":** `spec/ARCH_SPEC.md` § 5 describes `data.py` as providing *"lineage triples"* and § 6.4 references *"lineage-triple batching"* in `HAMPipeline`. The implementation stores `(parent, child)` pairs of shape `(P, 2)`, not triples. The code is internally consistent (field is named `lineage_pairs`, method is `extract_lineage_pairs`), but the spec terminology is misaligned. Either the spec should be updated to say "pairs" or the code should be extended to support triples `(ancestor, cell, descendant)`. **This discrepancy should be escalated to the spec maintainer.**

2. **`DataLoader` visibility:** `spec/ARCH_SPEC.md` § 5 lists `data.py` in the module tree but does not explicitly name `DataLoader` as a public class. The `__init__.py` exports only `BioDataset`. Spec and code agree that `BioDataset` is the primary public type, but the `DataLoader` class's intended API status is ambiguous.

3. **Velocity handling:** The spec does not describe the velocity fallback behavior (zero-filling when velocity is absent). This design choice has downstream implications for losses like `ZermeloAlignmentLoss` and `VelocityDirectionAlignmentLoss`, which consume `V` without checking whether it is genuine or zero-filled.
