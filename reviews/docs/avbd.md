**Reviewer:** Doc Reviewer Agent

**Source:** [src/ham/solvers/avbd.py](src/ham/solvers/avbd.py)

## Summary

Overall documentation quality: **needs work**.

The file implements the Augmented Vertex Block Descent (AVBD) boundary-value geodesic solver — one of two public solver interfaces (`spec/ARCH_SPEC.md` § 4.2) and a core component of the training pipeline. Both `AVBDSolver` and `Trajectory` are re-exported via [src/ham/solvers/\_\_init\_\_.py](src/ham/solvers/__init__.py) and used across six example scripts and three test modules.

The class-level docstring on `AVBDSolver` provides a reasonable one-paragraph summary and lists six attributes, but the **sole public method** `solve()` has **no docstring at all** — no `Args`, no `Returns`, no `Raises`, no mathematical context, no example pointers. The `Trajectory` NamedTuple (public API, exported in `__init__.py`) is entirely undocumented. The `SolverState` NamedTuple (internal) has inline field comments but no class docstring. There is no module-level docstring. Of 4 symbols defined at module scope, only 1 has any docstring, and that docstring is incomplete.

The file relies heavily on inline comments to explain algorithmic steps. While the comments are well-structured (numbered phases 1–6), they are not a substitute for formal docstrings and are invisible to `help()`, Sphinx, and IDE tooltips.

## Spec Alignment Notes

1. **`spec/ARCH_SPEC.md` § 4.1 — `GeodesicSolver` ABC**: The spec defines an abstract `GeodesicSolver` base class with `solve(self, metric, p_start, p_end, n_steps) -> Trajectory`. `AVBDSolver` does not inherit from this ABC — it extends `eqx.Module` directly. The `solve()` signature adds three extra parameters (`constraints`, `train_mode`, `key`) beyond the spec's interface. The documentation does not acknowledge this deviation.

2. **`spec/ARCH_SPEC.md` § 4.2 — Loss formulation**: The spec states the loss is $L = \sum E(x_i, v_i) + \text{ConstraintPenalties}$. The implementation's `local_action` computes a **two-segment** energy per vertex ($E(x_{i-1}, v_{in}) + E(x_i, v_{out})$), which double-counts each segment energy. This is standard for vertex-based relaxation (each segment contributes to two vertex losses) but differs from the spec's per-segment sum. The documentation should clarify this.

3. **`spec/MATH_SPEC.md` § 1.2 — Energy definition**: The spec defines $E(x, v) = \frac{1}{2} F^2(x, v)$. The inline comment at [avbd.py:97](src/ham/solvers/avbd.py#L97) says `# Discrete Finsler Energy: F(x, v)^2`, which incorrectly describes the quantity as $F^2$ rather than $E = \frac{1}{2}F^2$. The actual computation delegates to `metric.energy()`, which correctly computes $\frac{1}{2}F^2$ — so the code is correct, but the comment is misleading.

4. **`spec/ARCH_SPEC.md` § 4.2 — Differentiability**: The spec states the solver is "Fully differentiable w.r.t metric parameters." The implementation achieves this via `jax.lax.scan` unrolling in `train_mode=True`, but the `while_loop` branch (inference) is not differentiable. This distinction is not documented anywhere in the source file.

5. **Unused attributes**: `tol`, `energy_tol`, and `momentum` are documented in the class docstring as functional parameters but are dead code. The spec does not mention these — they appear to be aspirational features. The documentation should distinguish implemented from planned attributes.
