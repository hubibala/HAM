# Documentation Review: `src/ham/models/learned.py`

**Reviewer:** Doc Reviewer Agent  
**Date:** 2026-05-15  
**Spec Version:** 1.1.0 (Berwald Revision)  
**Source:** [src/ham/models/learned.py](src/ham/models/learned.py)

## Summary

Overall documentation quality: **needs work**.

The file defines seven public symbols — five metric classes (`NeuralRiemannian`, `NeuralRanders`, `PullbackRanders`, `PullbackRiemannian`, `DataDrivenPullbackRanders`) and two supporting modules (`PullbackGNet`, `KernelWindField`) — all re-exported via [src/ham/models/\_\_init\_\_.py](src/ham/models/__init__.py). These are central to every learned-metric workflow: six example scripts and the entire Weinreb experiment pipeline depend on them. `spec/ARCH_SPEC.md` § 3 and § 5 reference `models/learned.py` by name and list all five metric classes.

Every class has a class-level docstring, but they are uniformly **minimal** — one- or two-line summaries with no `Args`, no `Returns`, no `Raises`, no mathematical context, and no example pointers. The critical `__init__` methods (the only public API surface for most of these classes) are completely undocumented. `PullbackGNet` has **no docstring at all**. `KernelWindField` has a one-line docstring but documents none of its arguments or the kernel formula. There is no module-level docstring.

The file is missing from `spec/ARCH_SPEC.md` § 3 under a unified name — the spec calls the generic concept `LearnedFinsler` while the file uses five distinct class names. This naming discrepancy is not acknowledged anywhere.

## Spec Alignment Notes

1. **`spec/ARCH_SPEC.md` § 3 — `LearnedFinsler` does not exist.** The spec's metric hierarchy table lists `LearnedFinsler` as the class name for neural-network metrics. The actual implementation uses five distinct classes. The spec's § 5 module listing is accurate. **Recommendation:** Update `spec/ARCH_SPEC.md` § 3 table to reflect the actual class names, or add a module-level docstring note explaining the mapping.

2. **`spec/ARCH_SPEC.md` § 5 — listing is accurate.** The module listing correctly names all five metric classes (`NeuralRiemannian`, `NeuralRanders`, `PullbackRanders`, `PullbackRiemannian`, `DataDrivenPullbackRanders`). `KernelWindField` and `PullbackGNet` are not mentioned but are exported — consider adding them to § 5.

3. **`spec/MATH_SPEC.md` § 5 — Zermelo formula is not cited.** The Randers metric classes inherit the Zermelo parameterization $F(x,v) = (\sqrt{\lambda \|v\|_h^2 + \langle W,v \rangle_h^2} - \langle W,v \rangle_h) / \lambda$ from `Randers` in `zoo.py`, but none of the `learned.py` docstrings reference this formula or point to the spec section. An ML engineer would need to trace through two levels of inheritance to find the actual metric computation.

4. **`spec/MATH_SPEC.md` § 6 — Epsilon conventions.** The file uses at least two distinct $\epsilon$ values ($10^{-5}$ for Zermelo causality, $10^{-4}$ for metric tensor regularization) that are not aligned with the spec's $\epsilon$-regularization discussion in § 6.1 (which concerns $F_\epsilon(x,v) = \sqrt{F^2 + \epsilon^2}$). None of these distinctions are documented.

5. **Pullback metric not in MATH_SPEC.** The pullback construction $G(z) = J^T J$ is central to the bio application but has no dedicated section in `spec/MATH_SPEC.md`. The closest reference is in `spec/ARCH_SPEC.md` § 6 (Implementation Status). This is a gap in the mathematical specification rather than the code documentation, but the code docs should at minimum derive the formula inline.
