**Reviewer:** Doc Reviewer Agent

## Summary

Overall documentation quality: **needs work**.

The file implements two public classes — `WrappedNormal` and `GeometricVAE` — that together form the core bio-application interface described in `spec/ARCH_SPEC.md` § 5 (`bio/vae.py # GeometricVAE with Zermelo control dynamics`) and § 6, item 5. `GeometricVAE` is re-exported from `ham.bio` and used extensively across `examples/` (e.g., `weinreb_smoke_test.py`, `weinreb_vae.py`, `experiment_h2_directional.py`, `experiment_h3_discriminative.py`).

Both classes have class-level docstrings, but they are minimal one-liners. No method in the file has an `Args`, `Returns`, or `Raises` section. No mathematical formulations appear anywhere despite the code implementing non-trivial geometric operations (wrapped normal sampling via parallel transport and exponential map, KL divergence, Zermelo alignment, spray regularization). There is no module-level docstring and no references to spec sections or literature.

Key documentation gaps:

1. **No module docstring** — the file's role in the HAMTools bio pipeline is undiscoverable via `help()`.
2. **`WrappedNormal` is severely underdocumented** — a reader cannot understand the sampling procedure (tangent noise at origin → parallel transport to mean → exponential map) without reading the code line by line.
3. **`GeometricVAE.loss_fn`** packs four loss terms with hardcoded weights (`1e-4`, `0.1`, `1.0`) and returns a tuple, but none of this is documented.
4. **No mathematical notation** — critical for the mathematician audience.
5. **No links to spec sections or literature** (e.g., Nagano et al. for wrapped normal distributions).

## Spec Alignment Notes

1. **`spec/ARCH_SPEC.md` § 5** lists `bio/vae.py` as *"GeometricVAE with Zermelo control dynamics"*. The class docstring matches this description verbatim but adds no further detail. The spec also mentions `classifier_head` is a field on `GeometricVAE`, which matches the code but is entirely undocumented in the docstring.

2. **`spec/ARCH_SPEC.md` § 6, item 5** describes the bio application: *"GeometricVAE + DataDrivenPullbackRanders trained on Weinreb hematopoiesis data."* The `loss_fn` in `vae.py` hardcodes Hyperboloid-specific operations (`_minkowski_norm`, `_minkowski_dot`), but the spec recommends `EuclideanSpace` as the default (§ 6, item 6). The documentation does not mention this tension or guide the user toward the modular `ham.training.losses` API for non-Hyperboloid usage.

3. **`spec/MATH_SPEC.md` § 5** defines the Zermelo parameterization with the constraint $\|W\|_h < 1$. The `loss_fn` computes alignment with $W$ but does not reference this section or explain how the alignment loss relates to the Zermelo navigation problem.

4. **`spec/MATH_SPEC.md` § 2.1** defines the geodesic spray $G^i$. The `loss_fn` calls `self.metric.spray()` and `self.metric.inner_product()` but the docstring does not connect these to the spec's spray definition or explain why minimizing $\langle S, S \rangle$ encourages geodesic motion.

5. **`spec/MATH_SPEC.md` § 4.1** notes severe numerical instability when integrating curved manifolds (Sphere, Hyperboloid) with the VAE. Neither `WrappedNormal` nor `GeometricVAE` documents this known limitation, despite it being a critical operational concern for users.

6. **`WrappedNormal`** is not mentioned anywhere in either spec. This is a gap in the spec itself (out of scope for this review), but the class docstring should at minimum reference the relevant literature (Nagano et al., 2019; Mathieu et al., 2019) to establish mathematical grounding.
