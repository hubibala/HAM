# Documentation Review: `examples/demo_trajectories.py`

**Reviewer:** Doc Reviewer Agent  
**Date:** 2026-05-15

## Summary

Overall documentation quality: **needs work**.

The script is logically structured with numbered section headers and some inline comments, but it is missing a module-level docstring entirely, provides no mathematical context for the Randers/Zermelo framework it demonstrates, and leaves several parameter choices unexplained. A mathematician would not understand *why* specific solver parameters were chosen; an ML engineer would not know the geometric significance of the comparison being made.

## Spec Alignment Notes

1. **Randers construction** (`spec/ARCH_SPEC.md § 3.1`): The demo constructs a `Randers` metric with `h_net = lambda x: jnp.eye(3)` and a scaled Rossby–Haurwitz wind. The Zermelo navigation formula from `spec/MATH_SPEC.md § 5` is silently relied upon but never referenced in the script. A reader unfamiliar with Finsler geometry would not understand the connection between "wind + identity metric → asymmetric path cost."

2. **AVBDSolver** (`spec/ARCH_SPEC.md § 4.2`): The solver is used with non-default parameters (`beta=3.0` vs default `10.0`, `iterations=1000` vs default `20`). The script does not explain why these deviations from defaults are necessary for this specific problem.

3. **ExponentialMap** (`spec/ARCH_SPEC.md § 4.4`): The velocity scaling `* 40.0` implicitly relies on the BVP discretisation having `n_steps=40` segments. This coupling is fragile and undocumented.

4. **Comment accuracy** (`examples/demo_trajectories.py:85`): The comment "almost antipodal" contradicts the actual geometry — the start `[1,0,0]` and end `[0,0,1]` are orthogonal, not antipodal. This is an inaccurate description per `spec/MATH_SPEC.md § 4.1` (Sphere geometry).
