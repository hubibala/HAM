## Summary

Overall documentation quality: **needs work**.

The file defines the four concrete `FinslerMetric` subclasses in the library's geometric hierarchy (`Euclidean`, `Riemannian`, `Randers`, `DiscreteRanders`). These are all public API — re-exported via `src/ham/geometry/__init__.py` — and are exercised by six example scripts and seven test modules. Despite their central role, only `Randers` has a meaningful class-level docstring. The `Euclidean` and `Riemannian` class docstrings are single-line summaries with no `Args`, `Returns`, spec references, or examples. `DiscreteRanders` has a 5-word docstring. No `metric_fn` override in the file has any docstring at all. The module itself has no module-level docstring. The `Randers._get_zermelo_data` method has a partial docstring but omits argument/return documentation and shape information. Overall, 11 of 14 public/protected symbols lack adequate documentation.

## Spec Alignment Notes

1. **`spec/ARCH_SPEC.md` § 3 — Randers `metric_fn` description:** The ARCH_SPEC table describes the Randers `metric_fn` as `sqrt(v @ M(x) @ v) + dot(beta(x), v)` (the standard $\alpha + \beta$ form). The implementation uses the equivalent Zermelo navigation formula from `spec/MATH_SPEC.md` § 5. The `Randers` class docstring correctly identifies the Zermelo form, but the two spec documents describe different (mathematically equivalent) parameterizations without cross-referencing each other. **Recommended action:** Add a note to the `Randers` class docstring clarifying: *"The standard $\\alpha + \\beta$ form (ARCH_SPEC § 3) is equivalent to the Zermelo parameterization used here (MATH_SPEC § 5)."*

2. **`spec/MATH_SPEC.md` § 5 — sign convention:** The spec states *"We use the minus sign convention for $\\beta$ here to align with 'Headwind increases cost.'"* This convention is embodied in `metric_fn` at line 125 (`- W_dot_v`), but the docstring never mentions the sign convention. A differential geometer verifying the implementation will need this information.

3. **`spec/ARCH_SPEC.md` § 3 — `LearnedFinsler`:** The ARCH_SPEC table lists `LearnedFinsler` as a metric hierarchy entry. This class is not in `zoo.py` but in `models/learned.py`. This is correctly separated but the zoo.py module docstring (once created) should note that learnable variants live in `models/`.

4. **`spec/MATH_SPEC.md` § 4 — Geometric Hierarchy table:** The spec lists `Hyperboloid` as a geometry with "Minkowskian $\sqrt{\langle v, v\rangle_L}$" metric. The Hyperboloid manifold is in `surfaces.py`, not `zoo.py`, but no zoo.py class provides a Hyperboloid Finsler metric. The division of labor (manifold topology in `surfaces.py`, metric geometry in `zoo.py`) is not documented anywhere.

5. **`spec/MATH_SPEC.md` § 6.1 — Epsilon regularization:** The spec prescribes $F_\epsilon(x, v) = \sqrt{F^2(x, v) + \epsilon^2}$. The `Riemannian.metric_fn` uses `jnp.maximum(quad, 1e-12)` (clamp, not additive $\epsilon^2$). The `Randers.metric_fn` uses `jnp.maximum(discriminant, 0.0) + 1e-9` inside the sqrt (additive, closer to spec). The `Euclidean` uses `safe_norm` which uses `jnp.maximum(sum_sq, eps)`. These three different regularization strategies are undocumented and inconsistent with each other and with the spec.

6. **`spec/ARCH_SPEC.md` § 2.2 — Batch-First:** The spec states *"All operations assume a leading batch dimension `(B, ...)`."* The docstrings and type hints in `zoo.py` show single-point signatures `(D,)` with no mention of batch dimensions. The actual vmap batching is handled externally, but this is not documented.
