## Summary

Overall documentation quality: **needs work**.

This module implements `HAMPipeline` and `TrainingPhase`, the multi-phase declarative training orchestrator described in `spec/ARCH_SPEC.md` § 6.4. Both classes are re-exported via `src/ham/training/__init__.py` and are exercised in `examples/weinreb_smoke_test.py`, `examples/weinreb_experiment.py`, and `tests/test_pipeline.py`. Despite being the primary user-facing API for model training, the module has no module-level docstring, both classes have single-line docstrings with no argument/return/raises documentation, and the critical `fit()` method — a 90-line function with six parameters, multiple code paths, and subtle semantics around `requires_pairs` / `lineage_triples` — is entirely undocumented. An ML engineer cannot use this API without reading the source; a mathematician has no information about how losses are composed or how parameter freezing interacts with training phases.

## Spec Alignment Notes

1. **`spec/ARCH_SPEC.md` § 5** lists `pipeline.py` as `"HAMPipeline: multi-phase declarative training"`. The implementation matches this description, but the source file contains no reference back to the spec.

2. **`spec/ARCH_SPEC.md` § 6.4** states: *"`HAMPipeline` supports multi-phase training with per-phase parameter freezing, lineage-triple batching, and modular losses."* All three features are implemented but none are documented in the source. The terms "parameter freezing," "lineage-triple batching," and "modular losses" do not appear in any docstring.

3. **Dataset protocol** — The spec does not define a formal dataset interface. The `fit()` method implicitly requires `.X`, `.V`, `.labels`, `.lineage_pairs`, and optionally `.Traj_long`. This protocol should be documented either in the spec or in the source (or both). The `bio/data.py` module defines `BioDataset`, but `fit()` accepts any duck-typed object — this flexibility is undocumented.

4. **Batch data structure** — The spec mentions "lineage-triple batching" but does not specify the tuple layout passed to loss components. The three distinct tuple structures (`(x, v, labels)`, `(x_i, v_i, labels_i, x_j, x_k)`, `(x_start, x_end[, traj_long])`) constitute an implicit protocol that should be documented in either the spec or the source.

5. **Loss composition** — The spec states "modular losses" but does not specify the composition rule. The implementation uses additive sum with per-component weighting. This should be documented.
