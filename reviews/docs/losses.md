# Documentation Review: `src/ham/training/losses.py`

**Reviewer:** Doc Reviewer Agent  
**Date:** 2026-05-15  
**Spec Version:** 1.1.0 (Berwald Revision)  
**Source:** [src/ham/training/losses.py](src/ham/training/losses.py)

## Summary

Overall documentation quality: **needs work**.

The file implements 16 modular loss components and 3 module-level helper functions — collectively the largest public API surface in the `training/` subpackage. All 16 loss classes are re-exported via [src/ham/training/\_\_init\_\_.py](src/ham/training/__init__.py) and consumed by `HAMPipeline` (`spec/ARCH_SPEC.md` § 5). Every class has a one-line or short class-level docstring, which is good compared to other modules in the project. However, **none** of the classes document their `__call__` parameters (`model`, `batch`, `key`), return values, expected `batch` tuple layout, mathematical formulation, or link to the relevant spec sections. The base class `LossComponent` establishes a protocol but its docstring is a single sentence with no guidance for implementers or consumers.

The key documentation gaps are:

1. **No module docstring** — orientation for the 16 loss components and their role in the pipeline is absent.
2. **`batch` tuple layout is undocumented everywhere** — different losses expect `batch[0]`, `batch[1]`, `batch[2]` to be different things (single samples vs. pairs vs. trajectories), but this is never described.
3. **No mathematical formulations** — despite the classes implementing non-trivial Finslerian losses (Euler-Lagrange residuals, Randers action matching, Zermelo alignment), no LaTeX equations appear in any docstring.
4. **No `Args`/`Returns`/`Raises` sections** on any method.
5. **Private module-level helpers** (`_solve_and_integrate_impl`, `_solve_avbd_trajectory_impl`, `_solve_and_integrate`, `_solve_avbd_trajectory`) are partially documented — one has a docstring, one does not.

## Spec Alignment Notes

1. **`spec/ARCH_SPEC.md` § 5** lists `losses.py` as *"Modular loss components"* and § 4 (final paragraph) says `HAMPipeline` supports *"modular losses"*. The spec does not enumerate individual loss classes or their mathematical definitions. The spec is silent on batch layout conventions, making the module the sole source of truth — which reinforces the need for thorough module-level documentation.

2. **`spec/MATH_SPEC.md` § 1.2** defines $E = \frac{1}{2}F^2$. `FinslerActionMatchingLoss` (issue #25) documents $E = F^2$, contradicting the spec.

3. **`spec/MATH_SPEC.md` § 2.1** defines the Euler-Lagrange equations. `EulerLagrangeResidualLoss` correctly implements $R = \frac{d}{dt}(\partial_v L) - \partial_z L$ but the class docstring writes $R = d/dt (dL/dv) - dL/dz$ using non-standard partial/total derivative mixing. The spec uses $\frac{\partial E}{\partial v^i}$ notation.

4. **`spec/MATH_SPEC.md` § 5** (Zermelo) states the constraint $\|W\|_h < 1$. `WindThermodynamicLoss` computes $\|W\|_h^2$ which is the natural regularizer for this constraint, but the docstring neither cites the spec nor states the connection.

5. **`spec/MATH_SPEC.md` § 6.1** documents $F_\epsilon = \sqrt{F^2 + \epsilon^2}$. `EulerLagrangeResidualLoss` implements a different smoothing: $\sqrt{v^T H v + \epsilon^2}$ (smoothing only the Riemannian component). The relationship between the two is not documented (issue #20).

6. **`spec/ARCH_SPEC.md` § 2.2** defines `energy(x, v) = 0.5 * metric_fn(x, v)**2`. Multiple losses call `model.metric.energy()` without stating whether they expect $\frac{1}{2}F^2$ or $F^2$ — a consistent source of potential confusion (issue #25).
